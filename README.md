# iSecurify — GRC Platform

A full-featured **Governance, Risk, and Compliance (GRC)** platform for managing policies, frameworks, controls, audits, risks, vulnerabilities, evidence, and multi-tenant operations.

Built with **Next.js 16**, **TypeScript**, **Prisma ORM**, **MySQL**, and **shadcn/ui**.

## 🚀 Deployment — Ubuntu + Apache2 + Nginx + MySQL

iSecurify runs on a plain **Ubuntu 22.04 LTS** server — no Docker, no Coolify.
The deployment uses a four-layer stack:

```
┌────────────────────────────────────────────────────────────────────────┐
│                         Internet                                       │
└───────────────────────────┬────────────────────────────────────────────┘
                            │
                            ▼
┌────────────────────────────────────────────────────────────────────────┐
│  Apache2  (public edge, :80/:443, SSL via Let's Encrypt/certbot)       │
│  - Terminates TLS                                                       │
│  - Redirects HTTP → HTTPS                                              │
│  - Sets security headers (HSTS, CSP, X-Frame-Options, …)               │
└───────────────────────────┬────────────────────────────────────────────┘
                            │  ProxyPass http://127.0.0.1:8080/
                            ▼
┌────────────────────────────────────────────────────────────────────────┐
│  Nginx  (internal reverse proxy, 127.0.0.1:8080 only)                  │
│  - Static-asset caching (_next/static, brand icon)                     │
│  - Security headers (defense-in-depth)                                  │
│  - 50M upload limit                                                     │
└───────────────────────────┬────────────────────────────────────────────┘
                            │  proxy_pass http://127.0.0.1:3000
                            ▼
┌────────────────────────────────────────────────────────────────────────┐
│  Bun / Next.js  (systemd service, :3000)                               │
│  - Runs .next/standalone/server.js as the `isecurify` system user      │
│  - Reads env from /opt/isecurify/.env                                  │
│  - Logs to journald                                                     │
└───────────────────────────┬────────────────────────────────────────────┘
                            │  mysql2 driver
                            ▼
┌────────────────────────────────────────────────────────────────────────┐
│  MySQL 8.0  (localhost:3306)                                           │
│  - Database: isecurify                                                  │
│  - User: isecurify (random password, generated by deploy.sh)           │
└────────────────────────────────────────────────────────────────────────┘
```

> **Why two reverse proxies?** Apache2 is the public SSL terminator (it has the
> best certbot integration on Ubuntu). Nginx sits behind it as an internal
> layer for static-asset caching and security-header defense-in-depth. Both
> layers must be running for the app to be reachable from the internet.

### Prerequisites

- A fresh **Ubuntu 22.04 LTS** server (1 vCPU / 2 GB RAM minimum, 2 vCPU / 4 GB recommended)
- A **domain name** pointing at your server's public IP (A record, e.g. `isecurify.yourdomain.com → 1.2.3.4`)
- Ports **80** and **443** open in your firewall / cloud security group
- Root or sudo access
- (Optional) The repo uploaded to the server via rsync, or a git URL to clone from

### Step 1 — Upload the code to your server

If you have a git remote, the deploy script will clone it for you in Step 2 — skip ahead.

Otherwise, upload the project from your local machine:

```bash
# On your LOCAL machine — upload the project (rsync is fastest)
rsync -avz --exclude='node_modules' --exclude='.next' --exclude='.git' \
      ./isecurify/  ubuntu@YOUR_SERVER_IP:/tmp/isecurify/

# On the SERVER — move it into place
ssh ubuntu@YOUR_SERVER_IP
sudo mkdir -p /opt/isecurify
sudo rsync -a /tmp/isecurify/ /opt/isecurify/
sudo chown -R $USER:$USER /opt/isecurify
cd /opt/isecurify
```

### Step 2 — Run the deploy script

The deploy script does **everything**: installs Apache2 + Nginx + MySQL + Bun, creates the `isecurify` system user, creates the MySQL database + user (with a random password), syncs the Prisma schema, builds the Next.js standalone bundle, writes `/opt/isecurify/.env`, installs + enables the systemd service, installs the Nginx internal config, installs the Apache2 edge config, and issues an SSL certificate via certbot.

```bash
# Option A: deploy from a git repo (recommended)
sudo bash deploy/scripts/deploy.sh isecurify.yourdomain.com  https://github.com/your-org/isecurify.git

# Option B: deploy from the code already on the server (run from inside the repo)
cd /opt/isecurify
sudo bash deploy/scripts/deploy.sh isecurify.yourdomain.com
```

The script will prompt for the MySQL root password (press **Enter** if you haven't set one yet — fresh Ubuntu 22.04 installs use `auth_socket` and `sudo mysql` works without a password).

The full deploy takes 5–10 minutes depending on server speed and network latency (most of it is `bun install` + `next build` + certbot's SSL issuance).

### Step 3 — Seed demo data (first time only)

The deploy script does **not** seed the database by default (so re-running it on an existing install won't clobber your data). To seed the demo tenant (Acme Corporation) + sample users + sample frameworks/controls/audits/clients/certificates:

```bash
# 1. Flip the SEED_DB flag to true
sudo sed -i 's/SEED_DB=false/SEED_DB=true/' /opt/isecurify/.env

# 2. Restart so the app picks up the new flag
sudo systemctl restart isecurify

# 3. Wait 10-15 seconds for the seed to complete (watch the logs)
sudo journalctl -u isecurify -f
#   → look for "✅ Seed complete." then Ctrl-C

# 4. Flip the flag back to false (so the seed doesn't re-run on next restart)
sudo sed -i 's/SEED_DB=true/SEED_DB=false/' /opt/isecurify/.env

# 5. Restart one more time to pick up the corrected flag
sudo systemctl restart isecurify
```

### Step 4 — Verify the deployment

```bash
# 1. All three services should be "active (running)"
sudo systemctl status isecurify nginx apache2

# 2. The app should respond on HTTPS
curl -I https://isecurify.yourdomain.com
#   → HTTP/2 200  (or 307 redirect to /login)

# 3. SSL certificate should be valid
curl -vI https://isecurify.yourdomain.com 2>&1 | grep -E "(subject|issuer|expire)"

# 4. MySQL should have all the iSecurify tables
sudo mysql -u isecurify -p isecurify -e "SHOW TABLES;"
#   → expect ~20 tables including Client, Certificate, ReminderLog, User, …

# 5. Tail the app logs in real time
sudo journalctl -u isecurify -f
```

Open `https://isecurify.yourdomain.com` in your browser and log in with one of the [default credentials](#-default-login-credentials) below.

### Step 5 — Updates

To redeploy after pushing new code to your git repo:

```bash
sudo bash /opt/isecurify/deploy/scripts/update.sh
```

This runs `git pull` → `bun install` → `bunx prisma db push` → `bun run build` → `systemctl restart isecurify`. Takes 2–4 minutes typically.

If you deployed without git (Option B in Step 2), re-upload the code with rsync and run:

```bash
cd /opt/isecurify
sudo rsync -a --exclude='node_modules' --exclude='.next' --exclude='.git' \
           /tmp/isecurify/ ./
sudo bash deploy/scripts/update.sh
```

### Manual deployment (without the script)

If you prefer to deploy step-by-step instead of using `deploy.sh`, the four config files you need are:

| File | Install to | Purpose |
|------|-----------|---------|
| `deploy/systemd/isecurify.service` | `/etc/systemd/system/isecurify.service` | Runs `bun .next/standalone/server.js` as the `isecurify` user on port 3000 |
| `deploy/nginx/isecurify.conf` | `/etc/nginx/sites-available/isecurify` (+ symlink in `sites-enabled/`) | Internal reverse proxy on `127.0.0.1:8080` → `127.0.0.1:3000` |
| `deploy/apache/isecurify.conf` | `/etc/apache2/sites-available/isecurify.conf` (+ `a2ensite`) | Public edge on `:80/:443`, SSL termination, reverse proxy to `127.0.0.1:8080` |
| `.env.production.example` | copy to `/opt/isecurify/.env` and fill in real values | Database URL, NEXTAUTH_SECRET, app URL, etc. |

Then run, in order:

```bash
# 1. Install packages
sudo apt-get install -y curl git apache2 nginx mysql-server mysql-client \
                        certbot python3-certbot-apache openssl rsync

# 2. Install Bun
curl -fsSL https://bun.sh/install | bash
sudo ln -sf ~/.bun/bin/bun /usr/local/bin/bun

# 3. Create the isecurify user + /opt/isecurify
sudo useradd --system --create-home --home-dir /opt/isecurify --shell /bin/bash isecurify
sudo mkdir -p /opt/isecurify/uploads
sudo chown -R isecurify:isecurify /opt/isecurify

# 4. Create the MySQL database + user
sudo mysql <<'SQL'
CREATE DATABASE isecurify CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'isecurify'@'localhost' IDENTIFIED BY 'STRONG_PASSWORD_HERE';
GRANT ALL PRIVILEGES ON isecurify.* TO 'isecurify'@'localhost';
FLUSH PRIVILEGES;
SQL

# 5. Copy the code to /opt/isecurify (rsync from your local machine, or git clone)
# 6. Write /opt/isecurify/.env from .env.production.example (fill in the MySQL password + generate NEXTAUTH_SECRET)
# 7. Build the app
cd /opt/isecurify
sudo -u isecurify bun install
sudo -u isecurify bunx prisma generate --schema prisma/schema.mysql.prisma
sudo -u isecurify bunx prisma db push --schema prisma/schema.mysql.prisma --accept-data-loss
sudo -u isecurify bun run build
sudo cp -r .next/static .next/standalone/.next/
sudo cp -r public .next/standalone/
sudo chown -R isecurify:isecurify .next

# 8. Install + start the systemd service
sudo cp deploy/systemd/isecurify.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now isecurify

# 9. Install + reload Nginx (internal, port 8080)
sudo cp deploy/nginx/isecurify.conf /etc/nginx/sites-available/isecurify
sudo ln -sf /etc/nginx/sites-available/isecurify /etc/nginx/sites-enabled/isecurify
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

# 10. Install + reload Apache2 (edge, port 80/443)
sudo a2enmod proxy proxy_http ssl headers rewrite
sudo sed -i 's/isecurify.yourdomain.com/YOUR_REAL_DOMAIN/g' deploy/apache/isecurify.conf
sudo cp deploy/apache/isecurify.conf /etc/apache2/sites-available/isecurify.conf
sudo a2dissite 000-default
sudo a2ensite isecurify
sudo apache2ctl configtest && sudo systemctl reload apache2

# 11. Issue the SSL certificate
sudo certbot --apache -d YOUR_REAL_DOMAIN --redirect
sudo systemctl reload apache2
```

## ⚙️ Environment Variables

All environment variables are documented in [`.env.production.example`](.env.production.example) at the repo root. The deploy script (`deploy/scripts/deploy.sh`) generates `/opt/isecurify/.env` automatically with a random MySQL password and NEXTAUTH_SECRET — you only need to edit the template if you're deploying manually.

| Variable | Required | Description |
|----------|----------|-------------|
| `DATABASE_URL` | ✅ | MySQL connection string: `mysql://isecurify:PASSWORD@localhost:3306/isecurify` |
| `NEXTAUTH_SECRET` | ✅ | Random 32+ char string. Required by the `next-auth` package even though iSecurify uses custom scrypt-based session auth — the package checks for the env var at startup. |
| `NEXTAUTH_URL` | ✅ | Your app's public URL (with `https://`) |
| `NEXT_PUBLIC_APP_URL` | ✅ | Same as `NEXTAUTH_URL` (exposed to the client bundle) |
| `NODE_ENV` | ✅ | Set to `production` for deployed instances |
| `PORT` | No | Default: `3000` (the Bun/Next.js process port — Nginx proxies to this) |
| `HOSTNAME` | No | Default: `0.0.0.0` (bind to all interfaces so Nginx on localhost can reach it) |
| `UPLOAD_PATH` | No | Path for uploaded evidence files. Default: `/opt/isecurify/uploads` |
| `SEED_DB` | No | Set to `true` only on first deploy to create demo data, then back to `false` |

Generate a secure `NEXTAUTH_SECRET`:
```bash
openssl rand -base64 32
```

## 🗄 Database Schema

The platform uses **20 models** managed by Prisma ORM (schema at `prisma/schema.mysql.prisma`):

- **Tenant** — Multi-tenant organization
- **User** — Users with RBAC roles
- **Session** — Active login sessions
- **PasswordReset** — Password reset tokens
- **Framework** — Compliance frameworks (ISO 27001, SOC 2, GDPR, etc.)
- **Control** — Framework controls (importable via JSON/CSV/Word)
- **ControlAssignment** — Per-tenant control implementation status
- **Evidence** — Files/links proving control implementation
- **Checklist** — Compliance questionnaires
- **ChecklistItem** — Individual checklist questions
- **ChecklistAnswer** — User responses to checklist items
- **Vulnerability** — Security vulnerabilities with CVSS
- **Risk** — Risk register entries
- **Policy** — Organizational policies
- **PolicyApproval** — Policy approval workflow
- **Audit** — Audit engagements
- **AuditTask** — Tasks within an audit
- **AuditLog** — System audit trail
- **Notification** — In-app notifications
- **Client** — Client organizations (tenant-scoped, holds compliance certificates)
- **Certificate** — Compliance certificates (ISO 27001, SOC 2, etc.) linked to a Client, with audit dates, reminders, and pricing
- **ReminderLog** — Audit trail of manual/automatic reminder sends per Certificate

## ✅ Post-Deployment Checklist

- [ ] Application loads at `https://your-domain.com` (HTTP redirects to HTTPS automatically)
- [ ] Login works with `superadmin@isecurify.com` / `Admin@123` (or a tenant user from the [table above](#-default-login-credentials))
- [ ] Change default admin password immediately after first login
- [ ] SSL certificate is valid — check with `curl -vI https://your-domain.com 2>&1 | grep -E "(subject|issuer|expire)"`
- [ ] `sudo systemctl status isecurify nginx apache2` — all three report `active (running)`
- [ ] `sudo systemctl is-enabled isecurify` — reports `enabled` (starts on boot)
- [ ] MySQL database has all 20 tables — `sudo mysql -u isecurify -p isecurify -e "SHOW TABLES;"`
- [ ] Demo data seeded (Frameworks, Controls, Clients, Certificates all visible in their respective sidebar views)
- [ ] "Certificates Expiring Soon" widget on the dashboard shows 3+ items (the seed creates certs in the ≤30d / ≤60d / >60d buckets)
- [ ] File uploads work — try uploading a piece of evidence
- [ ] Firewall allows 80/443 — `sudo ufw status` (or your cloud provider's security group)
- [ ] Set up automatic MySQL backups (see [Automatic MySQL Backup](#automatic-mysql-backup-cron) below)
- [ ] Set up log rotation for `/var/log/apache2/isecurify_*.log` (logrotate is pre-installed on Ubuntu; default config rotates weekly)

## 🛠 Useful Commands

### Service Management — Bun/Next.js (systemd)
```bash
sudo systemctl start isecurify       # Start
sudo systemctl stop isecurify        # Stop
sudo systemctl restart isecurify     # Restart
sudo systemctl status isecurify      # Check status
sudo journalctl -u isecurify -f      # Follow logs (real-time)
sudo journalctl -u isecurify --since "1 hour ago"   # Recent logs
```

### Service Management — Nginx (internal reverse proxy, :8080)
```bash
sudo systemctl restart nginx
sudo systemctl reload nginx           # Reload without dropping connections
sudo nginx -t                         # Test config syntax
sudo tail -f /var/log/nginx/access.log
sudo tail -f /var/log/nginx/error.log
```

### Service Management — Apache2 (public edge, :80/:443)
```bash
sudo systemctl restart apache2
sudo systemctl reload apache2
sudo apache2ctl configtest            # Test config syntax
sudo apache2ctl -M                    # List loaded modules
sudo a2enmod proxy proxy_http ssl headers rewrite   # Enable required modules
sudo a2ensite isecurify               # Enable the iSecurify vhost
sudo a2dissite 000-default            # Disable the default vhost (frees :80)
sudo tail -f /var/log/apache2/isecurify_access.log
sudo tail -f /var/log/apache2/isecurify_error.log
```

### SSL Certificate Management (certbot / Let's Encrypt)
```bash
sudo certbot --apache -d isecurify.yourdomain.com           # Issue first cert
sudo certbot renew                                          # Renew all certs (dry-run: --dry-run)
sudo certbot renew --force-renewal -d isecurify.yourdomain.com   # Force renew one cert
sudo systemctl status certbot.timer                         # Check auto-renewal timer
```
> certbot installs a systemd timer that auto-renews certificates 30 days before expiry. No cron job needed.

### Database Management
```bash
# Connect to MySQL
sudo mysql -u isecurify -p isecurify

# Backup database (manual)
mysqldump -u isecurify -p isecurify > backup_$(date +%Y%m%d_%H%M%S).sql

# Restore database
mysql -u isecurify -p isecurify < backup_20250622.sql

# Push schema changes (after code update — usually handled by update.sh)
cd /opt/isecurify && sudo -u isecurify bunx prisma db push --schema prisma/schema.mysql.prisma
```

### Updates
```bash
# Easiest — use the update script (does git pull + bun install + prisma db push + build + restart)
sudo bash /opt/isecurify/deploy/scripts/update.sh

# Manual — same steps as update.sh, run by hand
cd /opt/isecurify
sudo -u isecurify git pull --ff-only
sudo -u isecurify bun install
sudo -u isecurify bunx prisma db push --schema prisma/schema.mysql.prisma --accept-data-loss
sudo -u isecurify bun run build
sudo cp -r .next/static .next/standalone/.next/
sudo cp -r public .next/standalone/

```

### Automatic MySQL Backup (Cron)
```bash
# Create backup script
sudo mkdir -p /opt/isecurify/backups

# Write the backup script (replace YOUR_DB_PASSWORD with the real password from /opt/isecurify/.env)
sudo tee /opt/isecurify/backup-db.sh > /dev/null <<'EOF'
#!/bin/bash
set -e
DATE=$(date +%Y%m%d_%H%M%S)
mysqldump -u isecurify -p"YOUR_DB_PASSWORD" isecurify | gzip > /opt/isecurify/backups/db_${DATE}.sql.gz
# Keep the last 30 days of backups; delete older ones
find /opt/isecurify/backups -name "*.sql.gz" -mtime +30 -delete
EOF

sudo chmod 700 /opt/isecurify/backup-db.sh   # only root can read (password is inside)
sudo chown root:root /opt/isecurify/backup-db.sh

# Run daily at 2 AM
echo "0 2 * * * root /opt/isecurify/backup-db.sh" | sudo tee /etc/cron.d/isecurify-backup
sudo chmod 644 /etc/cron.d/isecurify-backup
```

> To restore from a backup: `gunzip < /opt/isecurify/backups/db_YYYYMMDD_HHMMSS.sql.gz | sudo mysql -u isecurify -p isecurify`

## 🔧 Troubleshooting

| Issue | Likely Cause | Fix |
|-------|-------------|-----|
| **502 Bad Gateway** (browser) | Bun/Next.js process on :3000 is not running | `sudo systemctl status isecurify` — if failed, check `sudo journalctl -u isecurify -f` for the crash cause. Common: missing env var, DATABASE_URL wrong, `.next/standalone/server.js` missing (re-run `bun run build`). |
| **502 Bad Gateway** but `isecurify` is running | Nginx can't reach :3000, or Apache2 can't reach :8080 | `curl http://127.0.0.1:3000/api/auth/me` (should return JSON) → `curl http://127.0.0.1:8080/api/auth/me` (should also return JSON). If the first works but the second doesn't, `sudo nginx -t && sudo systemctl reload nginx`. If both work but browser still 502s, `sudo apache2ctl configtest && sudo systemctl reload apache2`. |
| **503 Service Unavailable** | Apache2 has no healthy backend (Nginx down) | `sudo systemctl status nginx` — restart if needed. Check `sudo tail -f /var/log/nginx/error.log`. |
| **Connection refused on :443** | Apache2 not running, or ports blocked by firewall | `sudo systemctl status apache2` → `sudo ufw status` (or your cloud security group). Ports 80 + 443 must be open. |
| **Redirect loop (HTTP → HTTPS → HTTP …)** | Apache2's `X-Forwarded-Proto` header isn't being set, so Next.js thinks it's still on HTTP | Verify `RequestHeader set X-Forwarded-Proto "https"` is in `/etc/apache2/sites-available/isecurify.conf`. Reload Apache2. |
| **SSL certificate error in browser** | certbot failed to issue, or cert expired | `sudo certbot certificates` (lists all certs + expiry) → `sudo certbot --apache -d your-domain.com --force-renewal`. Check `sudo systemctl status certbot.timer` for auto-renewal. |
| **certbot fails with "Connection refused"** | Port 80 not reachable from internet (DNS not propagated, or firewall blocks :80) | Verify DNS: `dig your-domain.com` should resolve to your server's public IP. Verify firewall: `sudo ufw status` must show 80 + 443 open. certbot's HTTP-01 challenge needs :80. |
| **`systemctl status isecurify` shows "failed"** | App crashed on startup | `sudo journalctl -u isecurify -n 100` — look for the stack trace. Most common causes: (1) `DATABASE_URL` wrong in `/opt/isecurify/.env`, (2) MySQL not running (`sudo systemctl status mysql`), (3) `.next/standalone/server.js` missing (re-run `bun run build`). |
| **`journalctl` shows "ECONNREFUSED 127.0.0.1:3306"** | MySQL is down or not listening | `sudo systemctl status mysql` → `sudo systemctl restart mysql`. If MySQL won't start, check `/var/log/mysql/error.log`. |
| **`apache2ctl configtest` fails** | Syntax error in vhost config, or required module not enabled | The error message names the file + line. Most common: `Invalid command 'ProxyPass'` → run `sudo a2enmod proxy proxy_http && sudo systemctl reload apache2`. |
| **`nginx -t` fails** | Syntax error in Nginx config | The error message names the file + line. Most common: `conflicting server name` → another site is already listening on the same port. Run `sudo ls /etc/nginx/sites-enabled/` and disable the conflict. |
| **Upload fails with "413 Request Entity Too Large"** | File exceeds the 50M limit set in both Nginx and Apache2 | Edit `client_max_body_size 50M;` in `/etc/nginx/sites-available/isecurify` AND `LimitRequestBody 52428800` in `/etc/apache2/sites-available/isecurify.conf` to match your new limit. Reload both. |
| **Permission denied on /opt/isecurify** | Files owned by root instead of isecurify user | `sudo chown -R isecurify:isecurify /opt/isecurify` (but NOT `/opt/isecurify/.env` if root needs to read it for cron backups — keep that as root:root with chmod 600). |
| **`prisma db push` fails with "Access denied"** | MySQL user lacks privileges | `sudo mysql -e "GRANT ALL PRIVILEGES ON isecurify.* TO 'isecurify'@'localhost'; FLUSH PRIVILEGES;"` |
| **Blank page after deploy, no errors in logs** | Build produced an empty standalone bundle | `ls -la /opt/isecurify/.next/standalone/` — should contain `server.js`. If empty, re-run `sudo -u isecurify bun run build` then `sudo cp -r .next/static .next/standalone/.next/ && sudo cp -r public .next/standalone/`. |
| **Data lost after restart** | `DATABASE_URL` accidentally points at SQLite (`file:./db/dev.db`) instead of MySQL | Check `/opt/isecurify/.env` — `DATABASE_URL` must start with `mysql://`. If you ran the seed against SQLite by mistake, switch to MySQL and re-seed. |
| **High memory usage / OOM kills** | Next.js build is memory-hungry; small VMs (1GB RAM) can OOM during `bun run build` | Add swap: `sudo fallocate -l 2G /swapfile && sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile && echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab`. |
| **Cron backup not running** | `/etc/cron.d/isecurify-backup` has wrong permissions or owner | `sudo chmod 644 /etc/cron.d/isecurify-backup` and ensure the file is owned by `root:root`. Check `sudo systemctl status cron`. |
| **Port 80 or 443 already in use by another service** | Default Apache site or another web server is squatting on the port | `sudo lsof -i :80` and `sudo lsof -i :443` to find the culprit → `sudo a2dissite 000-default` (if Apache) or `sudo systemctl stop <service>` then `sudo systemctl reload apache2`. |

## 💻 Local Development

```bash
# 1. Install dependencies
bun install

# 2. Setup .env (SQLite for local dev)
echo "DATABASE_URL=file:./db/dev.db" > .env

# 3. Push schema and seed
bunx prisma db push --schema prisma/schema.prisma
bunx prisma db seed

# 4. Start dev server
bun run dev
```

Open http://localhost:3000 — login with `superadmin@isecurify.com` / `Admin@123` (see [Default Login Credentials](#-default-login-credentials) above for the full list of seeded users).

> **Note:** Local dev uses the SQLite schema (`prisma/schema.prisma`). Production uses MySQL (`prisma/schema.mysql.prisma`). Both schemas are kept in sync — when you add a model to one, add it to the other.
